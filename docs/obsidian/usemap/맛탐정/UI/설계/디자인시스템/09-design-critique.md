# 9. Design Critique

맛탐정 디자인 시스템과 UI/UX 설계 전반에 대한 종합 크리틱.
Apple Design Director 관점에서 건설적·교육적·실행 가능한 피드백.

---

## 1. HEURISTIC EVALUATION (Nielsen's 10)

### H1. Visibility of System Status — 4/5

**잘된 점**: 좋아요·저장의 낙관적 업데이트, 스켈레톤 로딩, Toast 피드백 등 시스템 상태가 대부분 명확함.

**개선 필요**:
- 구독 후 "피드에 반영되기까지" 시간이 걸릴 수 있는데, 이 중간 상태 피드백이 부재함. "구독 완료! 새 맛집이 추가되면 피드에서 알려드릴게요" 같은 확인 메시지가 필요.
- 교차 검증 수 변화(출처 추가/삭제)에 대한 실시간 피드백이 정의되지 않음.

---

### H2. Match Between System and Real World — 4/5

**잘된 점**: "좋아요", "저장", "폴더", "구독" 등 메타포가 익숙함. 출처 배지의 컬러 코딩(유튜브=빨강)이 실제 브랜드와 일치.

**개선 필요**:
- "맛탐정"이라는 용어가 앱 이름인지, 큐레이터 역할인지, 탭 이름인지 세 가지로 중복 사용됨. 신규 사용자에게 혼란 가능. 탭 이름을 "탐색" 또는 "소스"로 분리하는 것을 검토할 것.
- "콜렉션"과 "폴더"가 혼용됨. 사용자 관점에서 하나의 용어로 통일해야 함.

---

### H3. User Control and Freedom — 4/5

**잘된 점**: 모달 drag-to-dismiss, 액션 시트의 "취소", 폴더 저장 시 체크박스 토글로 쉽게 해제 가능.

**개선 필요**:
- 좋아요·저장 실수 시 "되돌리기(Undo)"가 없음. Toast에 "되돌리기" 액션을 추가하면 사용자 안전망이 됨.
- 장소 상세 모달 안에서 소스 상세로 이동하면 depth가 깊어짐(모달 위 모달 또는 모달→페이지). 돌아오는 경로가 불명확할 수 있음.

---

### H4. Consistency and Standards — 3/5

**잘된 점**: 8px 그리드, 디자인 토큰 체계, 컴포넌트 variant 정의가 일관됨.

**개선 필요**:
- **홈 탭의 이중 탭 구조가 혼란스러움.** 하단 탭 바 4개 + 홈 내 상단 탭 3개(추천/소스/구독). "소스" 상단 탭과 하단 "맛탐정" 탭의 역할이 겹침. 이것이 디자인 시스템 전체에서 가장 큰 구조적 문제.
- 프로필의 9개 메뉴 칩이 플랫폼 표준을 따르지 않음. 9개 칩을 한 화면에 나열하면 인지 과부하. iOS의 Settings 패턴(List)이나 Segmented Control(3~4개)이 더 적합.
- PlaceCard의 Standard/Compact/Minimal 3가지 variant가 있지만, "언제 어떤 variant를 쓰는지" 규칙이 패턴 문서에서 충분히 명시되지 않음.

---

### H5. Error Prevention — 3/5

**잘된 점**: 삭제 전 확인 다이얼로그, 폴더 제거 시 "메모 보관" 안내.

**개선 필요**:
- 폴더 삭제가 소프트 삭제인지 하드 삭제인지 사용자에게 명확하지 않음. "삭제하면 복구할 수 없습니다" 경고 문구 필요.
- 로그인이 필요한 액션(좋아요, 저장, 구독 등)이 많은데, 비로그인 상태에서 이 버튼들이 보이지만 탭하면 로그인 모달이 뜸. 이것은 "유혹적 UI" 문제. 비로그인 상태에서 버튼을 다르게 표시하거나, 탭 시 왜 로그인이 필요한지 맥락을 설명해야 함.
- 초대 코드 24시간 만료 정책이 코드 생성 시에만 안내됨. 만료된 코드를 입력했을 때의 에러 메시지·복구 경로가 미정의.

---

### H6. Recognition Rather Than Recall — 4/5

**잘된 점**: 출처 배지가 항상 노출되어 "이 장소를 왜 저장했는지" 기억을 돕는 구조. 최근 검색어 표시. CollectionSaveSheet에서 최근 사용 폴더 상단 배치.

**개선 필요**:
- 폴더 내 장소에서 "이 장소를 어떤 소스에서 발견했는지"가 폴더 뷰에서 보이지 않을 수 있음. 폴더 리스트의 Minimal 카드에 출처 배지를 포함해야 Traceable 원칙이 관철됨.
- 프로필 메뉴 9개가 아이콘 없이 텍스트만이면 기억 부하가 높음.

---

### H7. Flexibility and Efficiency of Use — 3/5

**잘된 점**: Long press 컨텍스트 메뉴, 정렬(최신/거리), 2단계 저장 플로우(좋아요→폴더).

**개선 필요**:
- **파워 유저를 위한 단축 동작이 부족.** 예를 들어 PlaceCard에서 좌로 스와이프하면 바로 "폴더에 저장" 같은 제스처가 없음. 피드에서 대량의 장소를 빠르게 처리해야 하는 사용자에게 카드→상세→저장 플로우는 너무 길 수 있음.
- 검색에 "실시간 제안"이 (선택)으로 표기됨. 이것은 선택이 아니라 필수에 가까움. 자동완성 없는 검색은 2026년 기준으로 불편.
- 다중 선택(batch action)이 없음. 예: 좋아요 목록에서 여러 장소를 선택해 한 번에 폴더로 이동.

---

### H8. Aesthetic and Minimalist Design — 5/5

**잘된 점**: "별점 없음"이라는 과감한 결정이 미니멀리즘의 정수. 출처 배지만으로 신뢰를 전달하는 구조는 정보 밀도를 낮추면서 핵심 차별점을 강조하는 우수한 설계. backdrop-blur 금지, transition 최소화 등 성능과 미학의 균형도 적절함.

---

### H9. Help Users Recognize, Diagnose, and Recover from Errors — 2/5

**잘된 점**: Empty state마다 CTA가 있어 "다음에 무엇을 할지" 안내함.

**개선 필요**:
- **에러 시나리오 정의가 전반적으로 부족.** 현재 문서에서 정의된 에러: "장소 정보를 불러올 수 없어요 + 재시도", "네트워크 연결 확인" 2가지뿐.
- 다음 에러 케이스가 미정의:
  - 저장/좋아요 API 실패 시 낙관적 업데이트 롤백 피드백
  - 폴더 생성 실패 (네트워크/서버 에러)
  - 구독 토글 실패
  - 이미지 로딩 실패 (broken image placeholder)
  - 검색 시 서버 타임아웃
  - OAuth 로그인 실패/취소
- 에러 Toast와 Alert의 사용 기준이 불명확. "언제 Toast, 언제 Alert?"

---

### H10. Help and Documentation — 2/5

**잘된 점**: 온보딩 3단계 슬라이드가 핵심 가치를 전달.

**개선 필요**:
- **"교차 검증"이라는 핵심 개념을 어디서 학습하는가?** 온보딩 3번째 슬라이드에서 한 번 언급되지만, 앱 내에서 "출처가 3개 이상이면 무엇이 좋은지" 인-앱 교육이 없음.
- 출처 배지를 처음 보는 사용자가 "빨간 배지 vs 파란 배지"의 의미를 어떻게 아는가? 툴팁이나 첫 노출 시 코치 마크(tooltip overlay)가 필요.
- FAQ가 랜딩 페이지에만 있고, 앱 내 도움말/가이드가 설계되지 않음.
- "맛탐정이 되는 법" 가이드가 앱 내에 없음. 공개 폴더를 만들면 큐레이터가 된다는 흐름이 자연스럽게 드러나야 함.

## 2. VISUAL HIERARCHY ANALYSIS

### 사용자가 가장 먼저 보는 것 — 의도와 일치하는가?

| 화면 | 의도된 1st | 실제 1st | 일치 여부 |
|------|-----------|----------|-----------|
| Home | PlaceCard 이미지 | PlaceCard 이미지 | ✅ |
| PlaceDetail | 이미지 갤러리 | 이미지 갤러리 | ✅ |
| SourceDetail | 소스 프로필(Avatar+이름) | 소스 프로필 | ✅ |
| Profile | Avatar + 닉네임 | Avatar + 닉네임 | ✅ |
| Search | 검색 입력 필드 | 검색 입력 필드 | ✅ |

**문제**: PlaceDetail에서 출처 하이라이트가 "네 번째"로 인지됨. 이것이 맛탐정의 핵심 차별점인데, 이미지·제목·지역보다 뒤에 있음. 의도적 결정(Interest Hierarchy)이라고 설명되어 있지만, 출처가 스크롤 없이 보이는지(above the fold) 검증이 필요.

### CTA Hierarchy

- Primary CTA가 화면당 1개 규칙은 우수.
- **문제**: PlaceActionRow에서 "좋아요, 저장, 네이버, 더보기" 4개 액션이 동등한 시각적 무게를 가짐. 하지만 맛탐정의 비즈니스 목표상 "저장"이 가장 중요한 전환 액션. "저장" 버튼의 시각적 무게를 높이는 것을 고려해야 함.

### White Space

- Content Density 표에서 화면별 밀도를 정의한 것은 우수.
- **문제**: 프로필 화면이 "Low"로 정의되어 있지만, 9개 메뉴 칩 + Stats + 콘텐츠 리스트를 담으면 실제로는 Medium 이상이 될 수 있음.

## 3. TYPOGRAPHY AUDIT

### Font 선택

- **Pretendard Variable**: 한국어 가독성이 우수하고, 가변 폰트로 성능 최적화에 적합. 좋은 선택.
- **문제**: 기존 프로젝트의 tailwind.config에는 `IBM Plex Sans KR`이 설정되어 있고, 디자인 시스템에서는 Pretendard를 지정함. 마이그레이션 계획이 필요.

### Type Scale

- 10단계(Display~Caption) 스케일이 명확하고, 각 용도가 정의되어 있음.
- **문제**: Title 2(18px)와 Title 3(17px)의 차이가 1px뿐. 화면에서 시각적으로 구분하기 어려울 수 있음. Title 3를 16px(Body와 동일)로 낮추거나 Title 2를 19px로 올리는 것을 검토.

### Line Length

- 모바일 375px 기준, 본문 16px에 margin 16px×2 = 343px 콘텐츠 너비. 한글 기준 약 20~22자/줄. 모바일에서 적절함.
- **문제**: 태블릿·데스크톱에서 max-width 512px로 제한하고 있어 줄당 글자 수가 30~32자. 한글 기준 적절하나, 영문이 섞이면 길어질 수 있음.

### Contrast

- 본문(#171717 on #FAFAFA): 17.4:1 — 우수.
- Caption(#737373 on #FFFFFF): 4.6:1 — AA 통과하나 여유 없음.
- **문제**: Error 텍스트(#EF4444 on #FFFFFF): 4.0:1 — AA 미통과(일반 텍스트). Error 색상을 #DC2626으로 어둡게 하거나, Error 배경 위에서만 사용하도록 제한해야 함.

## 4. COLOR ANALYSIS

### Brand Personality 적합성

- Primary(보라 #7C3AED)가 "신뢰 + 미니멀"에 적합한가? 보라는 보통 "창의성, 프리미엄"을 연상시킴. 맛탐정의 "신뢰"라는 감성과는 약간 거리가 있을 수 있음.
- **대안 검토**: 딥 블루(#2563EB)가 "신뢰"에 더 적합할 수 있으나, 네이버(녹색)/카카오(노랑)/인스타(보라~핑크) 사이에서 차별화된 색상 선택이라면 보라도 합리적.

### 의미 있는 색상 사용

- 출처 배지의 컬러 코딩(유튜브=빨강, 커뮤니티=파랑, 폴더=보라, 인스타=핑크)은 실제 브랜드 색상과 연결되어 직관적. 우수.
- Semantic Colors(Success/Warning/Error/Info)가 명확하게 정의됨.
- **문제**: 좋아요 활성 상태가 "accent-rose"이고, Error도 빨강 계열. 두 색상의 의미적 충돌 가능. 좋아요를 Primary 색상으로 변경하거나, accent-rose와 error의 명도 차이를 더 벌리는 것을 검토.

### Dark Mode

- 1차 라이트만, 토큰으로 준비 후 추후 지원 — 현실적이고 올바른 판단.
- Dark Mode 등가색이 정의되어 있어 전환 시 수월할 것.

## 5. USABILITY CONCERNS

### 5.1 Cognitive Load

**높은 인지 부하 영역**:
- **프로필 메뉴 9개**: 최근, 좋아요, 콜렉션, 저장, 방문, 리뷰, 구독, 구독자, 분석. 너무 많음. 사용자가 "저장"과 "좋아요"의 차이, "콜렉션"과 "저장"의 차이를 즉시 이해할 수 있는가?
- **홈 탭 내 상단 탭 3개**: 추천/소스/구독. "소스"와 하단 탭 "맛탐정"의 차이가 불명확.
- **장소 상세 탭 3개**: 메뉴/리뷰/출처. "리뷰"가 설계 철학상 소극적 기능인데 탭으로 노출하는 것이 모순.

**권장**: 프로필 메뉴를 5개 이하로 그룹핑. "좋아요+저장"을 하나의 "내 장소"로 합치고, "구독+구독자"를 하나의 "소셜"로 합치는 등.

### 5.2 Interaction Clarity

- 출처 배지가 클릭 가능한지 카드 레벨에서 시각적으로 불명확. 배지를 탭하면 소스 상세로 이동하는데, "탭 가능"이라는 어포던스(underline, chevron 등)가 없음.
- "네이버" 버튼이 PlaceActionRow에 있지만, 외부 링크인지 앱 내 이동인지 구분이 안 됨. 외부 링크 아이콘(↗)을 추가해야 함.

### 5.3 Mobile Touch Targets

- 44×44px 최소 터치 영역 원칙이 명시되어 있음. 좋음.
- **문제**: 출처 배지(padding 4×8, font 12px)는 터치 영역이 약 20×28px. 카드 내에서 배지를 개별 탭하기 어려움. 배지를 개별 탭 대상으로 할 것인지, 배지 그룹 전체를 탭 → 출처 목록으로 이동할 것인지 명확히 결정해야 함.

### 5.4 Form Usability

- 폴더 생성 폼: 제목·설명·권한 입력. 순서와 필수/선택 구분이 미정의.
- 권한 선택(public/private/hidden/invite)이 4가지인데, 신규 사용자가 이 차이를 이해하기 어려움. 기본값(public)을 명시하고, 각 옵션에 한 줄 설명을 추가해야 함.

## 6. STRATEGIC ALIGNMENT

### 비즈니스 목표 부합 여부

| 목표 | 부합 | 근거 |
|------|------|------|
| 사용자 유입(AWARENESS) | ✅ | 온보딩, 출처 차별점 명확 |
| 저장 전환(CONVERSION) | ⚠️ | 저장 플로우가 2~3단계. 단축 필요 |
| 재방문(RETENTION) | ⚠️ | 피드 구독이 핵심이나, 구독 유도가 약함 |
| 큐레이터 확보 | ❌ | "맛탐정 되기" 유도 동선이 앱 내에 없음 |

### 사용자 목표 부합 여부

| 목표 | 부합 | 근거 |
|------|------|------|
| 출처 있는 맛집 발견 | ✅ | 출처 배지, 교차 검증 |
| 맥락과 함께 저장 | ✅ | 폴더+메모+출처 |
| 지역별 꺼내 쓰기 | ⚠️ | 필터는 있으나, "내 저장 목록"에서의 지역 필터가 명시되지 않음 |

### Value Proposition 명확성

- "출처가 다르면, 발견이 다르다" — 강력하고 명확.
- **문제**: 이 메시지가 앱 내에서 어디에 노출되는가? 온보딩 이후 앱 안에서는 이 핵심 메시지를 재확인할 곳이 없음.

### 경쟁 차별화

- "별점 없음 + 출처 교차 검증"은 명확한 차별점. 우수.
- **문제**: "출처 개수"가 아직 적은 초기 단계에서는 교차 검증의 가치가 드러나기 어려움. 소스 1개뿐인 장소가 대다수일 때의 UX가 미설계.

## 8. REDESIGN DIRECTION

### Alternative A: "3탭 + 통합 탐색" 구조

홈 상단 탭과 하단 탭의 중복을 해결하는 구조.

```text
하단 탭 3개:
[탐색]  [피드]  [프로필]

"탐색" 탭 (기존 홈+맛탐정 통합):
+----------------------------------+
| [위치] [검색]                    |
+----------------------------------+
| [추천]  [유튜브]  [커뮤니티]      |
| [인스타] [폴더]                  |  ← 소스 타입별 진입
+----------------------------------+
| 🔥 교차 검증 TOP                 |
| [PlaceCard] [PlaceCard] ...      |
+----------------------------------+
```

**장점**:
- 하단 탭 3개로 단순화 (인지 부하 감소)
- "소스"와 "맛탐정"의 중복 제거
- 소스 타입별 진입점이 명확

**단점**:
- "구독 피드"가 "피드" 탭으로 분리되면서 구독의 중요성이 약해질 수 있음
- 홈에서 소스 타입 칩이 많아지면 혼잡할 수 있음

### Alternative B: "4탭 + 홈 단순화" 구조

하단 탭 4개를 유지하되, 홈의 상단 탭을 제거하는 구조.

```text
하단 탭 4개:
[홈]  [검색]  [피드]  [프로필]

"홈" (상단 탭 없음):
+----------------------------------+
| [위치: 강남]           맛탐정     |
+----------------------------------+
| 🔥 교차 검증 TOP 5               |
| [PlaceCard 가로 스크롤]          |
+----------------------------------+
| 📺 유튜브 인기 소스               |
| [SourceCard 가로 스크롤]         |
+----------------------------------+
| 📁 인기 폴더                     |
| [FolderCard 가로 스크롤]         |
+----------------------------------+

"검색" (기존 탐색+검색 통합):
+----------------------------------+
| [🔍 검색]                        |
+----------------------------------+
| [지역 칩] [카테고리 칩] [필터]    |
+----------------------------------+
| [PlaceCard 리스트]               |
+----------------------------------+
```

**장점**:
- 홈이 대시보드처럼 다양한 콘텐츠를 한눈에 보여줌
- 검색이 독립 탭이 되어 "탐색" 행동의 진입점이 명확
- "소스 탐색"이 홈 섹션 → 소스 상세로 자연스럽게 연결

**단점**:
- 홈 화면 콘텐츠가 많아져 스크롤이 길어질 수 있음
- "맛탐정" 브랜딩이 탭에서 사라짐

> **Critique Director's Note**: Alternative B를 추천함. 이유: (1) 홈 상단 탭 제거로 인지 부하 감소, (2) 검색이 독립 탭이면 "탐색 → 발견 → 저장" 플로우가 명확해짐, (3) 홈의 가로 스크롤 섹션이 다양한 소스를 소개하는 데 적합. 다만 홈의 스크롤 길이가 지나치지 않도록 섹션 수를 3~4개로 제한해야 함.

## 총평

맛탐정의 디자인 시스템은 **핵심 철학(출처 기반 발견, 별점 없음)이 UI 전반에 일관되게 반영되어 있다는 점에서 매우 우수함.** "별점이 있는 자리에 출처 배지를 넣는다"는 단순한 구조 변경이 전체 경험을 차별화하는 것은 Apple이 "기본값을 바꿔서 새로운 경험을 만드는" 접근과 일맥상통함.

다만 **네비게이션 구조의 중복(홈 상단 탭 vs 하단 맛탐정 탭), 에러 시나리오의 부재, 핵심 개념(교차 검증)의 학습 경로 미설계**는 런칭 전에 반드시 해결해야 할 과제임. 디자인의 완성도보다 "사용자가 핵심 가치를 이해하는가?"에 더 집중해야 할 시점.

가장 좋은 디자인은 사용자가 "이 앱은 다른 맛집 앱과 뭐가 다르지?"라고 묻기 전에, 출처 배지를 보는 순간 "아, 출처가 보이는구나"라고 스스로 깨닫게 만드는 것. 현재 설계는 그 방향으로 가고 있지만, 학습 보조 장치(코치 마크, 툴팁)가 추가되면 완성도가 크게 올라갈 것.
