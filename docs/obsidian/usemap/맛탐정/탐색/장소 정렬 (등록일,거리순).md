# 맛탐정 탐색 - 장소 정렬 기능

## 1. 왜 필요한가: 논리적 필요성

### 1.1 사용자 시나리오와 문제

맛탐정은 **장소 목록**을 다루는 화면이 여러 개 존재함. 각 화면의 맥락이 다름:

| 맥락 | 사용자 의도 | 기대 행동 |
|------|-------------|-----------|
| 피드 | "지금 내 주변에서 뭘 먹을까?" | 가까운 순으로 보여줘야 함 |
| 폴더 | "이 맛집 리스트에서 지금 가까운 곳은?" | 등록일순만으론 부족함 |
| 피처(유튜브/네이버 등) | "이 채널/폴더 맛집 중 가까운 곳은?" | 동일한 니즈 |

**핵심 인사이트**: 장소 목록은 **시간축(최신순)**과 **공간축(거리순)** 두 가지로 해석 가능함. 사용자가 "지금 먹으러 갈 곳"을 찾을 때는 공간축이 더 중요함.

### 1.2 기존 한계

- **FeedPage/HomePage**: 최신순·거리순 지원하나, UI·로직이 각각 중복 구현됨.
- **FolderDetailPage**: 맛탐정 폴더 내 장소는 **등록일순 고정**. 사용자가 "이 폴더에서 가까운 곳"을 찾을 수 없음.
- **FeatureDetailPage**: 네이버 폴더, 유튜브 채널, 커뮤니티/소셜 지역 등 5종 피처 모두 **정렬 옵션 없음**.

→ **일관성 부재**: 같은 "장소 목록"인데 화면마다 정렬 능력이 다름. 사용자 인지 부담 증가.

### 1.3 논리적 결론

1. **기능 패리티**: 장소 목록이 있는 모든 화면에서 동일한 정렬 옵션 제공해야 함.
2. **코드 일관성**: 중복 제거로 유지보수성·버그 감소.
3. **UX 일관성**: "최신순/거리순 + 위치 설정" 패턴을 한 번 학습하면 모든 화면에서 동일하게 사용 가능함.

## 2. 의도와 아이디어 콘셉트

### 2.1 핵심 콘셉트: "장소 목록 = 시간×공간"

```
                    시간축 (최신순)
                         ^
                         |
    "언제 추가됐는가"   |   "지금 어디가 가까운가"
                         |
    ---------------------+---------------------> 공간축 (거리순)
                         |
                         |
```

- **최신순**: "최근에 추가된 맛집" — 콘텐츠 발견·탐색에 적합.
- **거리순**: "지금 내 위치에서 가까운 맛집" — 실제 방문 결정에 적합.

두 축을 토글로 전환 가능하게 하면, 사용자가 맥락에 맞게 선택 가능함.

### 2.2 설계 원칙

1. **단일 진실 소스**: 정렬 상태·위치 선택 로직을 한 곳에서 관리.
2. **레이어 분리**: UI(SortToggle) ↔ 상태(usePlaceSort) ↔ API ↔ DB.
3. **서버 사이드 정렬**: 무한 스크롤·페이지네이션 환경에서 클라이언트 정렬은 불가능. Haversine 거리 계산은 DB에서 수행함.

### 2.3 Feed vs Folder/Feature 차이

| 구분 | Feed | Folder/Feature |
|------|------|----------------|
| 거리 필터 | 5km 이내만 | 제한 없음, 전체 대상 |
| 이유 | "지금 먹으러 갈 수 있는" 범위 제한 | "이 리스트 전체에서 가까운 순" 정렬 |

### 5.2 공통 훅: usePlaceSort

**파일**: `src/shared/hooks/usePlaceSort.ts`

```tsx
interface UsePlaceSortOptions {
  useUrlSync?: boolean;   // true: URL ?sort= 동기화 (Feed/Home)
  defaultSort?: 'recent' | 'distance';
}

// 반환값
{
  sortBy, selectedLocation, isLocationSheetOpen, setIsLocationSheetOpen,
  handleSortByRecent, handleSortByDistance, handleLocationSelect

```

- `useUserLocations`로 최근 위치 초기값 설정
- 거리순 클릭 시 위치 미설정이면 `LocationSettingSheet` 오픈
- `handleLocationSelect`: 위치 선택 시 거리순으로 전환
- Feed/Home: `useUrlSync: true` → URL `?sort=recent|distance` 동기화
- Folder/Feature: 로컬 상태

### 5.3 DB 함수 업데이트

**파일**: `docs/sql/092_add_distance_sort_to_all_place_apis.sql`

| 함수 | 대상 | 추가 파라미터 | 반환 추가 |
|------|------|----------------|-----------|
| v1_get_folder_places | 맛탐정 폴더 | p_sort_by, p_user_lat, p_user_lng | distance_meters |
| v3_get_places_by_naver_folder | 네이버 폴더 | 동일 | distance_meters |
| v3_get_places_by_youtube_channel | 유튜브 채널 | 동일 | distance_meters |
| v3_get_places_by_community_region | 커뮤니티 지역 | 동일 | distance_meters |
| v3_get_places_by_social_region | 소셜 지역 | 동일 | distance_meters |
| v3_get_places_by_region | 통합 지역 | 동일 | distance_meters |

**Haversine 거리 계산** (PostgreSQL, 미터 단위):

```sql
(6371 * acos(LEAST(1.0, GREATEST(-1.0, 
  cos(radians(p_user_lat)) * cos(radians(pl.y::double precision)) * 
  cos(radians(pl.x::double precision) - radians(p_user_lng)) + 
  sin(radians(p_user_lat)) * sin(radians(pl.y::double precision))
)))) * 1000
```

### 5.4 API/Query 레이어 변경

**Folder** (`src/entities/folder/`):

- `queries.ts`: `useFolderPlaces(folderId, visitedOnly, sortOptions)` 시그니처
- `types.ts`: `FolderPlace`에 `distance_meters?: number` 추가

**Place** (`src/entities/place/`):
